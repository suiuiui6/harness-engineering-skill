# 多模态知识管理平台 — 从零搭建完整规划

## 一、项目概览

你要构建的是一个 **多模态文档智能知识管理平台**，核心链路为：

```
文档上传 (PDF/Word/PPT) → 文档解析 → 实体/关系抽取 → 知识图谱构建 → 自然语言问答
```

这是一个典型的 **RAG（检索增强生成）+ 知识图谱（GraphRAG）** 混合系统。下面按照"自底向上"的方式做完整规划。

---

## 二、技术选型建议

### 2.1 文档解析层

不同格式需要不同的解析策略：

| 文档格式 | 推荐方案 | 备选方案 |
|---------|---------|---------|
| **PDF** | MinerU（结构化提取）或 PyMuPDF + Unstructured | Marker, LlamaParse |
| **Word (.docx)** | python-docx 或 Unstructured | Mammoth（转 Markdown）|
| **PPT (.pptx)** | python-pptx 或 Unstructured | LibreOffice 命令行转换 |

**关键决策点**：
- 如果文档以**中文为主**、对表格和公式有高要求，优先考虑 MinerU。
- 如果追求**轻量快速集成**，Unstructured 库覆盖面最广，一个库通吃三种格式。
- 如果文档以**扫描件/图片型 PDF** 为主，必须引入 OCR（Tesseract / PaddleOCR）。

### 2.2 实体与关系抽取层

这是系统智能化的核心，有两类路线可选：

**路线 A：基于大模型（推荐起步方案）**
- 使用 LLM（如 GPT-4o / Claude / 通义千问）进行信息抽取
- 设计 Prompt 模板，让 LLM 输出结构化 JSON（实体列表 + 三元组关系列表）
- 配合 LangChain / LlamaIndex 的抽取链统一调度
- 优点：开发快、对复杂语义理解好；缺点：成本高、速度慢、依赖 API

**路线 B：基于传统 NLP（高吞吐场景）**
- 命名实体识别：spaCy / HanLP（中文）
- 关系抽取：OpenNRE / DeepKE
- 优点：速度快、成本低、可离线；缺点：中文效果有限、需要训练/微调

**推荐策略**：先用 LLM 路线快速验证可行性，后期对高频文档类型训练专用抽取模型降低成本。

### 2.3 知识图谱存储层

| 方案 | 适用场景 | 特点 |
|------|---------|------|
| **Neo4j** | 通用图数据库首选 | Cypher 查询语言成熟、生态丰富、可视化强 |
| **ArangoDB** | 需要同时支持图和文档存储 | 多模型数据库，省去额外数据库 |
| **NetworkX + JSON** | 小规模/MVP 阶段 | 纯 Python、零部署、快速验证 |

**推荐策略**：MVP 阶段用 NetworkX + JSON 文件快速迭代；生产阶段迁移到 Neo4j。

### 2.4 向量存储与检索层

GraphRAG 模式下，除了图查询还需要语义检索：

| 方案 | 特点 |
|------|------|
| **ChromaDB** | 轻量、Python 原生、适合 MVP |
| **Milvus** | 高性能、分布式、适合生产 |
| **Qdrant** | Rust 编写、性能优秀、API 友好 |
| **Pinecone / Weaviate** | 云托管、免运维 |

**推荐策略**：MVP 用 ChromaDB 快速搭建；生产阶段根据规模选 Milvus 或 Qdrant。

### 2.5 大模型与问答层

这里有两层 LLM 需求：

| 环节 | LLM 角色 | 推荐模型 |
|------|---------|---------|
| 实体/关系抽取 | 理解文本、输出结构化数据 | GPT-4o / Claude Sonnet / 通义千问-turbo |
| 自然语言问答 | 理解问题、查询知识图谱、生成回答 | GPT-4o / Claude / DeepSeek |

**技术框架选择**：
- **LangChain**：生态成熟、组件丰富、GraphRAG 有现成实现
- **LlamaIndex**：对知识图谱 + RAG 的组合支持更好，内置 GraphRAG 能力
- **Dify / FastGPT**：如果需要低代码可视化编排

**推荐**：如果你熟悉 Python 开发，用 LangChain 或 LlamaIndex；如果需要快速 Demo，用 Dify。

### 各模块职责

**前端层**：
- 拖拽上传区（支持 PDF/Word/PPT）
- 上传进度展示
- 知识图谱可视化（ECharts / D3.js / vis-network）
- 对话式问答界面（Chat UI）
- 文档管理和检索面板

**API 网关层**（FastAPI）：
- `POST /documents/upload` — 文档上传与解析
- `GET /documents/{id}/graph` — 获取图谱数据
- `POST /qa/ask` — 自然语言问答
- `GET /entities/search` — 实体搜索
- WebSocket 支持流式问答输出

**文档解析 Pipeline**：
- 格式检测与路由（根据文件头判断类型）
- PDF → MinerU / PyMuPDF 逐页解析
- Word → python-docx 提取段落、表格、图片
- PPT → python-pptx 提取每页文本和备注
- Chunk 策略：按段落/章节切分，保留上下文边界
- 每个 Chunk 做 Embedding 写入向量库

**知识抽取与图谱构建**：
- 每个文档 Chunk 送 LLM 做实体抽取 + 关系抽取
- Prompt 示例结构：
  ```
  请从以下文本中抽取实体和关系，以 JSON 格式返回：
  {
    "entities": [{"name": "...", "type": "人物/组织/概念/..."}, ...],
    "relations": [{"source": "...", "target": "...", "relation": "..."}, ...]
  }
  ```
- 跨 Chunk 实体对齐（同名实体合并、指代消解）
- 写入图数据库（节点 = 实体，边 = 关系，附加源文档和位置元数据）

**问答引擎**：
- 接收用户自然语言问题
- 第一步：问题理解与实体识别（抽取问题中的实体关键词）
- 第二步：多路召回
  - 图查询：在图谱中检索相关实体和关系子图
  - 向量检索：在向量库中检索语义相似的 Chunk
- 第三步：上下文融合与答案生成（LLM 基于召回的图结构 + 原文片段生成答案）
- 支持追问和上下文延续（多轮对话）

## 四、项目搭建分阶段路线图

### 阶段 0：环境准备（1-2 天）

1. 确定技术栈的最终选型（从上面的推荐中做决策）
2. 搭建 Python 项目骨架：
   ```
   knowledge-platform/
   ├── backend/           # FastAPI 后端
   │   ├── api/           # 路由与接口
   │   ├── services/      # 业务逻辑
   │   ├── models/        # 数据模型
   │   └── config/        # 配置文件
   ├── parsers/           # 文档解析模块
   ├── extraction/        # 实体关系抽取模块
   ├── graph/             # 图谱构建与存储模块
   ├── qa/                # 问答引擎
   ├── frontend/          # 前端代码
   ├── data/              # 数据文件（gitignore）
   └── docs/              # 项目文档
   ```
3. 配置开发环境：uv / Poetry 管理依赖，`.env` 管理 API Key
4. 初始化 Git 仓库，建立 commit 规范

### 阶段 1：文档解析 MVP（3-5 天）

**目标**：上传一份 PDF/Word/PPT，系统能输出结构化文本内容。

1. 实现文件上传 API（FastAPI + file upload）
2. 集成 Unstructured 库 / python-docx / python-pptx
3. 实现统一的解析接口：
   ```python
   def parse_document(file_path: str, file_type: str) -> ParsedDocument:
       # 返回结构化段落、表格、图片信息列表
   ```
4. 写一个简单的 HTML/Streamlit 页面上传文件并查看解析结果
5. 用 2-3 份真实文档做调试，验证解析质量

**验收标准**：一份 10 页 PDF 能在 30 秒内完成解析，段落/表格无遗漏，中文不乱码。

### 阶段 2：实体抽取与图谱构建 MVP（5-7 天）

**目标**：从解析后的文本中抽取实体和关系，生成可视化知识图谱。

1. 设计 LLM 抽取 Prompt（实体类型定义、关系类型定义、JSON Schema）
2. 实现抽取 Service，对接 LLM API（OpenAI / Claude 兼容接口）
3. 将 Chunk 级抽取结果汇总，做实体对齐和去重
4. 用 NetworkX 构建图数据，导出为 JSON 供前端使用
5. 前端用 ECharts / vis-network 做图谱可视化（力导向图）
6. 处理大文档的批处理（限流、重试、进度追踪）

**验收标准**：一篇 3000 字文章抽取后，实体数量 > 20，关系数量 > 15，图谱可视化可交互（拖拽、缩放、点击查看详情）。

### 阶段 3：知识问答 MVP（5-7 天）

**目标**：用户可以用自然语言提问，系统基于知识图谱给出回答。

1. 实现双路检索：
   - 图检索：基于问题中的实体名匹配图数据库中的节点，提取 k-hop 邻居子图
   - 向量检索：问题 Embedding → ChromaDB 相似搜索 → 返回相关 Chunk
2. 设计问答 Prompt：
   ```
   基于以下知识图谱信息和相关文档片段，回答用户问题。
   知识图谱子图（JSON 格式）：...
   相关文档片段：...
   用户问题：...
   请给出准确、有依据的回答，并标注信息来源。
   ```
3. 实现对话历史管理（多轮对话上下文）
4. 前端实现 Chat UI（类似 ChatGPT 的对话界面）
5. 回答中附带来源引用（文档名 + 段落位置）

**验收标准**：针对已上传文档的内容提问，回答准确率 > 80%，能明确指出信息来源。

### 阶段 4：优化与生产化（7-14 天）

1. **性能优化**：
   - 文档解析异步化（Celery / FastAPI BackgroundTasks）
   - LLM 调用批量化（合并多个 Chunk 一起抽取）
   - Embedding 缓存（相同 Chunk 不重复计算）
   - 图谱索引优化

2. **功能增强**：
   - 支持图片中的文字 OCR（PaddleOCR / Tesseract）
   - 支持文档中的图片内容理解（多模态 LLM）
   - 实体属性补全（如人物的职位、时间等信息）
   - 时间轴视图（按时间线展示实体关系变化）

3. **数据库正式化**：
   - 从 NetworkX 迁移到 Neo4j
   - 从 ChromaDB 迁移到 Milvus / Qdrant
   - 引入 PostgreSQL 存储用户、文档元数据等业务数据
   - MinIO 作为文件存储

4. **工程化**：
   - Docker 容器化（每个服务独立容器）
   - docker-compose 一键启动
   - 单元测试与集成测试覆盖
   - CI/CD Pipeline（GitHub Actions）
   - API 文档自动生成（FastAPI 自带 Swagger）

## 五、关键技术风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| **PDF 表格/公式解析不准** | 抽取质量差 | 先用 MinerU 专项测试，必要时代入公式 OCR（LaTeX-OCR） |
| **LLM 抽取幻觉** | 实体关系不准确 | 引入置信度评分，低分结果人工审核；Prompt 中要求 LLM 逐句核对原文 |
| **实体对齐失败** | 图谱节点重复、边断开 | 使用文本相似度（Cosine）合并相似实体名称，配合 LLM 做消歧 |
| **大文档解析慢** | 用户体验差 | 后台异步处理 + 进度推送（WebSocket），前端显示处理进度条 |
| **LLM API 成本过高** | 不可持续 | 对高频文档类型微调专属小模型（如 Qwen 系列）；缓存相似 Chunk 的抽取结果 |
| **多模态内容丢失** | 图片中的信息未抽取 | 在解析阶段保留图片路径，抽取阶段引入多模态 LLM 做图片描述 |

## 六、推荐起步方案（最小可行路径）

如果你想在 **1-2 周内看到一个可运行的 Demo**，建议用以下最简组合起步：

```
前端:       Streamlit（纯 Python，零前端开发成本）
文档解析:    Unstructured（一库通吃 PDF/Word/PPT）
向量库:      ChromaDB（pip install 即用）
图谱:        NetworkX + JSON（纯 Python，无需部署）
LLM:        OpenAI GPT-4o API（或国内通义千问）
框架:       LangChain（封装 LLM 调用和 Chain 逻辑）
```

这个组合的优点是：**所有组件都是 Python 原生或 pip install 即用，零外部服务部署**，让你先把完整链路跑通，验证核心价值，再逐步替换为生产级组件。
